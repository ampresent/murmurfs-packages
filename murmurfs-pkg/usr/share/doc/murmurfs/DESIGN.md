# MurmurFS — Design Document

## Overview

MurmurFS is a FUSE-based virtual filesystem designed for AI agents. Instead of storing concrete file contents, it stores **stacks of intent** — each layer describes *what the agent plans to do*, not *what the file contains*. Files exist in a perpetual "fuzzy" state until explicitly synced, at which point an LLM synthesizes all intent layers into a final, concrete file.

The name "murmur" evokes the low, indistinct whispers of intent — not yet clear, not yet decided, but layered and building toward something.

## Core Metaphor

Think of a palimpsest — an ancient manuscript where old text was scraped off and new text written over it, but traces of the old always bled through. MurmurFS is the same: every revision leaves a trace, every intent layer is preserved, and the final form emerges only when you decide to materialize it.

```
Agent's view (fuzzy)              After sync (concrete)
┌─────────────────┐              ┌─────────────────────┐
│ v1: 做认证模块   │              │ import jwt           │
│ v2: 只要 JWT     │   ──sync──>  │                      │
│ v3: 加 refresh   │              │ def login(user):     │
│ v4: 考虑 OAuth2  │              │     ...              │
└─────────────────┘              │ def refresh(token):  │
                                  │     ...              │
                                  └─────────────────────┘
```

## Goals

1. **Reduce agent context usage** — Plans/intents are shorter than full file contents
2. **Defer precision** — Agent doesn't need to decide exact implementation details upfront
3. **Preserve reasoning history** — Every layer captures *why* a decision was made
4. **Enable composability** — Multiple agents can contribute intent layers to the same file
5. **Batch LLM costs** — Squash (cheap) and sync (expensive) are separate operations

## Architecture

### Filesystem Layout

Each project using MurmurFS has two directories:

```
project/
├── murmur/          # Virtual mount point (FUSE) — the "fuzzy" view
│   ├── src/
│   │   └── auth.py  # Looks like a file, actually an intent stack
│   └── lib/
│       └── utils.py
├── real/            # Synced concrete files (actual source code)
│   ├── src/
│   │   └── auth.py  # Real, runnable code (generated by sync)
│   └── lib/
│       └── utils.py
└── .murmurfs/       # Metadata and intent storage
    └── manifest.yaml
```

### Intent Layers

Each file's intent is stored as a stack of layers. Each layer contains:

```yaml
- id: v1
  timestamp: 2026-04-16T01:55:00Z
  summary: "实现用户认证模块，JWT方式"
  full: "实现用户认证模块，支持 JWT 和 session 两种方式，
         包含登录、登出、token 验证中间件"
```

- **summary**: One-line description shown to the agent on `read()`
- **full**: Complete intent description, used during squash/sync

When an agent reads `auth.py` from the murmur mount, it sees:

```
v1: 实现用户认证模块，JWT方式
v2: 砍掉session，只保留bearer token
v3: 增加refresh token逻辑，过期时间15分钟
```

### Operations

#### `read(path)` → Intent Summary

Returns the stack of intent summaries for the given file. If the file has been synced, returns the actual file contents instead.

#### `write(path, intent)` → Append Layer

Appends a new intent layer to the file's stack. Does NOT modify any real file.

```
echo "考虑加上 OAuth2 支持" >> auth.py   →  appends v4
```

#### `squash(path)` → Compress Layers

Invokes the LLM to compress multiple intent layers into a single, richer layer. Low cost — only modifies the summary metadata, doesn't generate code.

**Before squash:**
```
v1: 实现用户认证模块
v2: 砍掉session
v3: 加refresh token
v4: 考虑OAuth2
```

**After squash:**
```
v1: 实现JWT认证模块，含refresh token（15分钟过期），预留OAuth2扩展点
```

#### `sync(path)` → Generate Real File

Invokes the LLM to read the (possibly squashed) intent layers and generate the actual file content. Writes to `real/` directory. High cost.

#### `merge(path)` → Resolve Concurrent Intents

When multiple agents append conflicting intents to the same file, all layers are preserved. At squash/sync time, the LLM resolves semantic conflicts:

```
Agent A: "加 OAuth2 支持"
Agent B: "改用 session 认证"

Merge result (LLM decides): "支持 OAuth2 和 session 双认证，
                              通过配置切换"
```

Conflicting intents are never silently dropped — the LLM must actively reason about them.

### Directory Metadata

The filesystem itself is fuzzy. `ls` returns a combination of:
1. **Real files** that exist on disk (synced)
2. **Intended files** described in the manifest but not yet materialized

```yaml
# .murmurfs/manifest.yaml
directories:
  src/:
    description: "核心业务逻辑"
    intended:
      - payment.py    # "v1: 支付模块，支持支付宝和微信"
  lib/:
    description: "通用工具函数"
```

When an agent runs `ls src/`, MurmurFS shows both `auth.py` (real) and `payment.py` (intended from metadata). Reading `payment.py` returns its intent stack even though no real file exists yet.

### Sync Triggers

Sync can be triggered by:

1. **Manual** — Agent explicitly calls `murmurfs sync <file>`
2. **Threshold** — Layer count exceeds a configured limit (e.g., 10 layers)
3. **Agent judgment** — Agent decides "this is ready to materialize"
4. **Project-wide** — `murmurfs sync --all` syncs everything

### Squash Triggers

Same as sync triggers, but cheaper. Squash can run much more frequently:
- After every N writes
- On a timer
- Before sync (auto-squash then sync)

## LLM Integration

### Squash Prompt

```
Given these intent layers for a file, compress them into a single
comprehensive intent description that captures all decisions and reasoning:

Layer 1: {summary_v1}
  Detail: {full_v1}
Layer 2: {summary_v2}
  Detail: {full_v2}
...

Output a single layer with:
- summary: one-line description
- full: complete intent with all key decisions preserved
```

### Sync Prompt

```
Given this intent description, generate the file contents.

File: {path}
Intent: {full_intent}
Project context: {project_description}
Related files (summaries): {related_file_summaries}

Generate the complete file. Follow the project's conventions and style.
```

### Merge Prompt

```
These intent layers were written by different agents and may conflict.
Analyze them and produce a unified intent that resolves conflicts
reasonably. If a true conflict exists that can't be auto-resolved,
flag it.

{all_layers_with_agent_ids}
```

## Data Format

### manifest.yaml Schema

```yaml
version: 1
project:
  name: "my-project"
  description: "A web application backend"

files:
  <path>:
    description: "Human-readable file purpose"
    layers:
      - id: <string>
        timestamp: <ISO-8601>
        summary: <string>
        full: <string>
        author: <agent-id>
    squashed: <bool>
    synced: <bool>
    sync_target: <path>  # relative to project root
    last_sync: <ISO-8601 | null>

directories:
  <path>:
    description: "Directory purpose"
    intended: [<filename>, ...]
```

### Intent Layer Storage

Intent layers are stored inline in `manifest.yaml` for simplicity in the prototype. For production, layers could be stored as individual files under `.murmurfs/layers/<file-path>/v1`, `v2`, etc.

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Intent format | Plain text | Simple, LLM-friendly, no parsing overhead |
| Metadata storage | Sidecar YAML file | Human-readable, version-controllable, simple |
| Squash vs Sync separation | Two distinct operations | Squash is cheap, sync is expensive — separate concerns |
| Conflict resolution | LLM decides, always preserve all layers | Never silently drop intent; let LLM reason about conflicts |
| Post-sync behavior | Keep intent history | Enables undo/audit; sync doesn't destroy the "why" |
| Directory listing | Merge real + intended | Agent sees the full "vision" of the project, not just what exists |
| FUSE mount visibility | Agent knows it's MurmurFS | Transparency over magic — agent should understand the system |

## Future Considerations

- **Multi-project linking** — Intent layers referencing files in other MurmurFS projects
- **Semantic search over layers** — "Show me all files where I considered adding OAuth"
- **Layer branching** — Fork an intent stack to explore alternative designs
- **Cost tracking** — Track LLM tokens spent on squash vs sync per file
- **Partial sync** — Sync only a subset of layers (e.g., v1-v3, ignore v4)
